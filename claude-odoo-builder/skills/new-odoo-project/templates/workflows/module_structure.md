# Odoo Module Structure Reference

## Objective
Define the standard Odoo module directory layout so every module we build is consistent, maintainable, upgrade-compatible, and easy for any developer to navigate.

## Directory Responsibilities

### `__manifest__.py` (required)
Module metadata. Always include:
- `name`, `version`, `summary`, `category`
- `depends` ‚Äî list of required modules (`website` for website snippets)
- `data` ‚Äî XML/CSV files loaded on install (views, security, data)
- `assets` ‚Äî frontend bundle registration (Odoo 15+)
- `installable: True`, `auto_install: False`, `license: 'LGPL-3'`

If `security/ir.model.access.csv` exists, add it to `data` **before** any view files so access rules are in place when views load.

### `__init__.py` (required)
At the root: import the `models` subpackage if it exists (`from . import models`).
At `models/__init__.py`: import each model file (`from . import my_model`).
Leave empty if the module has no Python models.

### `models/` ‚Äî Business logic and ORM
One file per logical model (e.g., `product_template.py`, `sale_order.py`). Each file defines a class inheriting from `models.Model`, `models.TransientModel`, or `models.AbstractModel`.

When to create: any time the module stores data in the database beyond standard Odoo fields.

### `views/` ‚Äî UI definitions
XML files defining forms, lists, kanban views, menus, and actions. For website modules, also contains snippet templates and editor registration XPaths.

### `security/ir.model.access.csv` ‚Äî Access control
Controls which user groups can Read/Write/Create/Delete records of each model. Required whenever `models/` contains at least one new model class.

Header row (always present, even if no rules added yet):
```
id,name,model_id:id,group_id:id,perm_read,perm_write,perm_create,perm_unlink
```

Example rule:
```
access_my_model_user,my.model user,model_my_model,base.group_user,1,1,1,0
```

Omitting this file when new models exist causes `AccessError` on install.

### `wizard/` ‚Äî Transient business actions
Use for multi-step operations that don't persist data long-term (e.g., import dialogs, batch actions, confirmation popups). Models here inherit from `models.TransientModel`. Records are auto-deleted after a configurable period.

When to create: only if the module needs a popup or multi-step user interaction that isn't handled by a standard form view.

### `reports/` ‚Äî QWeb reports
XML templates for PDF or HTML reports (invoices, delivery slips, etc.). Registered via `ir.actions.report` records, usually defined in the same XML file.

When to create: only if the module generates printable documents.

### `data/` ‚Äî Initial data
XML files loaded once at install time. Use for default configuration records, demo data stubs, or master data (e.g., product categories, email templates, sequences).

When to create: any time the module needs records to exist on a fresh install.

### `i18n/` ‚Äî Translations
A `.pot` template file (generated by Odoo's `--i18n-export`) and per-language `.po` files. Odoo picks these up automatically on language install.

When to create: when the module will be used in multiple languages or distributed publicly.

### `static/src/` ‚Äî Frontend assets
All JavaScript, SCSS, images, and fonts. Subdirectories: `js/`, `scss/`, `img/`, `fonts/`. Assets are registered in `__manifest__.py` under the `assets` key (Odoo 15+) or via `ir.asset` records.

## Naming Conventions

- **Module technical name:** `snake_case`, globally unique (e.g., `website_hero_slider`, `sale_custom_discount`)
- **Model class names:** `PascalCase`, mirrors the model name (e.g., `class SaleCustomDiscount(models.Model)`)
- **Model `_name`:** `dot.separated.lowercase` (e.g., `_name = 'sale.custom.discount'`)
- **XML `id` attributes:** unique within the module; prefix with the module name when referencing from other modules
- **SCSS root selector:** `.s_{technical_name}` for snippets ‚Äî all rules scoped under this class
- **Security CSV `id` column:** `access_{model_snake}_{group_snake}` (e.g., `access_my_model_user`)

---

## Key Rules

1. **Security first in `data`:** list `security/ir.model.access.csv` before any view XML in `__manifest__.py['data']`
2. **Scope all SCSS:** every rule in `static/src/scss/` must be nested under the module's root class (`.s_{name}`) to prevent style bleed
3. **No hardcoded IDs in templates:** snippet templates must not use `id=""` attributes ‚Äî multiple instances can live on one page
4. **`o_editable` marks editable regions:** any text the website editor user should be able to modify needs this class
5. **`img-fluid` on every image:** required for responsive layout in Odoo's editor
6. **One model per file in `models/`:** keeps files short and diffs clean
7. **TransientModel for wizards:** never use `models.Model` for data that doesn't need to persist

## Relation to Tools

- **`scaffold_snippet.py`** generates the full directory structure above automatically. Run it first before writing any custom code.
- **`push_page.py`** and **`push_mailing.py`** operate on page/mailing arches, not module files ‚Äî they are not relevant to module development.
- **`migrate_to_production.py`** handles CSS and page migrations, not module deployment. For module deployment see `create_snippet.md`.
